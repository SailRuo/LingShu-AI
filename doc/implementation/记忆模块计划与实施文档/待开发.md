## 一、分段阶梯衰减算法

### 1.1 背景

当前记忆活跃度计算使用的是简单指数衰减：

```java
// MemoryServiceImpl.java
private double calculateActivityScore(LocalDateTime activatedAt, LocalDateTime now) {
    long daysSinceActivation = ChronoUnit.DAYS.between(activatedAt, now);
    return Math.exp(-daysSinceActivation / 30.0); // 30天半衰期
}
```

这种方案虽然简单，但不符合人类记忆的衰减规律。认知科学研究表明，记忆的衰减是非线性的：近期记忆衰减快，远期记忆衰减慢。

### 1.2 目标

实现分段阶梯衰减算法，更精确地模拟记忆衰减规律：

| 时间窗口 | 衰减系数 | 说明 |
|---------|---------|------|
| 0-6小时 | 0.96 | 短期记忆，衰减较慢 |
| 6-24小时 | 0.84 | 中期记忆，衰减加速 |
| 1-7天 | 0.65 | 过渡期，衰减明显 |
| 7-30天 | 0.34 | 长期记忆，衰减趋缓 |
| 30天以上 | 0.15 | 远期记忆，趋于稳定 |

### 1.3 实现方案

```java
private double calculateActivityScore(LocalDateTime activatedAt, LocalDateTime now) {
    long hoursSinceActivation = ChronoUnit.HOURS.between(activatedAt, now);
    long daysSinceActivation = ChronoUnit.DAYS.between(activatedAt, now);
    
    if (hoursSinceActivation <= 6) {
        return Math.pow(0.96, hoursSinceActivation / 6.0);
    } else if (daysSinceActivation == 0) {
        return 0.96 * Math.pow(0.84, (hoursSinceActivation - 6) / 18.0);
    } else if (daysSinceActivation <= 7) {
        return 0.96 * 0.84 * Math.pow(0.65, (daysSinceActivation - 1) / 6.0);
    } else if (daysSinceActivation <= 30) {
        return 0.96 * 0.84 * 0.65 * Math.pow(0.34, (daysSinceActivation - 7) / 23.0);
    } else {
        return 0.96 * 0.84 * 0.65 * 0.34 * Math.pow(0.15, (daysSinceActivation - 30) / 30.0);
    }
}
```

### 1.4 预期效果

| 场景 | 当前实现 (指数衰减) | 分段衰减 |
|------|-------------------|---------|
| 1小时前 | 0.99 | 0.99 |
| 12小时前 | 0.96 | 0.91 |
| 1天前 | 0.93 | 0.81 |
| 3天前 | 0.86 | 0.62 |
| 7天前 | 0.79 | 0.42 |
| 30天前 | 0.37 | 0.14 |
| 90天前 | 0.05 | 0.03 |

- **影响场景**: 记忆生命周期维护、事实状态流转、检索排序
- **风险评估**: 低，仅影响活跃度计算，不改变数据结构

## 二、DEPENDS_ON 逻辑依存关系

### 2.1 背景

当前记忆系统支持三种事实关系：

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| `RELATED_TO` | 事实之间相关 | "喜欢喝茶" ↔ "喜欢读书" |
| `SUPERSEDES` | 新事实替代旧事实 | "住在北京" → "住在上海" |
| `CONTRADICTS` | 事实之间冲突 | "喜欢猫" ↔ "对猫过敏" |

但缺少 `DEPENDS_ON` 关系，导致当某个事实被标记为冲突或替代时，依赖该事实的其他记忆无法联级更新，产生"僵尸记忆"。

### 2.2 目标

引入 `DEPENDS_ON` 关系，支持记忆链条的联级冲突处理和生命周期同步。

**典型场景**：

```
事实A: "我在北京工作" (observedAt: 2026-01-15)
    ↓ DEPENDS_ON
事实B: "我租的房子在北京朝阳区" (observedAt: 2026-01-20)
    ↓ DEPENDS_ON
事实C: "我每天通勤1小时" (observedAt: 2026-01-25)

当用户说"我搬到上海了":
- 事实A 被标记为 superseded
- 事实B 自动标记为 conflicted (因为依赖A)
- 事实C 自动标记为 conflicted (因为依赖B)
```

### 2.3 实现方案

#### 2.3.1 数据模型扩展

在 `FactNode` 实体中增加字段：

```java
@Node("Fact")
public class FactNode {
    // ... 现有字段 ...
    
    private Long dependsOnFactId;  // 依赖的父事实ID
    private String dependencyType; // STRONG(强依赖) / WEAK(弱依赖)
}
```

**依赖类型说明**：

| 类型 | 说明 | 联级行为 |
|------|------|---------|
| `STRONG` | 强依赖，父事实失效则子事实必然失效 | 联级标记为 conflicted |
| `WEAK` | 弱依赖，父事实失效不影响子事实 | 仅记录日志，不联级处理 |

#### 2.3.2 依存关系识别

在事实提取阶段，通过 LLM 识别依存关系：

```java
public interface FactDependencyEvaluator {
    @SystemMessage("""
            你是灵枢的记忆依存关系评估模块。
            
            给定一个新提取的事实，判断它是否依赖于已有的某个事实。
            
            【依存关系定义】
            - STRONG: 如果父事实失效，子事实必然失效
              示例: "我租的房子在北京" DEPENDS_ON "我在北京工作"
            
            - WEAK: 父事实失效，子事实可能仍然有效
              示例: "我喜欢北京烤鸭" DEPENDS_ON "我在北京" (弱依赖)
            
            - NONE: 无依存关系
            
            【返回格式】
            {
                "dependsOnFactId": 123,
                "dependencyType": "STRONG/WEAK/NONE",
                "reason": "因为租房地点依赖于工作地点"
            }
            """)
    DependencyResult evaluate(
        @V("newFact") String newFact,
        @V("existingFacts") List<String> existingFacts
    );
}
```

#### 2.3.3 联级更新逻辑

在 `MemoryServiceImpl` 中增加联级更新方法：

```java
private void cascadeUpdateRelations(FactNode invalidatedFact, String newStatus) {
    // 1. 查找所有依赖该事实的子事实
    List<FactNode> dependents = factRepository.findByDependsOnFactId(invalidatedFact.getId());
    
    for (FactNode dependent : dependents) {
        // 2. 根据依赖类型决定处理方式
        if ("STRONG".equals(dependent.getDependencyType())) {
            dependent.setStatus("conflicted");
            dependent.setLastActivatedAt(LocalDateTime.now());
            factRepository.save(dependent);
            
            systemLogService.info(String.format(
                "联级更新: 事实[%d]因依赖事实[%d]被标记为%s，自动标记为conflicted",
                dependent.getId(), invalidatedFact.getId(), newStatus
            ), "MEMORY");
            
            // 3. 递归处理子事实的依赖链
            cascadeUpdateRelations(dependent, "conflicted");
        } else if ("WEAK".equals(dependent.getDependencyType())) {
            systemLogService.info(String.format(
                "弱依赖提示: 事实[%d]弱依赖于事实[%d](已%s)，无需联级处理",
                dependent.getId(), invalidatedFact.getId(), newStatus
            ), "MEMORY");
        }
    }
}
```

#### 2.3.4 Neo4j 关系维护

       "MATCH (a:Fact) WHERE id(a) = rel.sourceId " +
       "MATCH (b:Fact) WHERE id(b) = rel.targetId " +
       "MERGE (a)-[r:DEPENDS_ON]->(b) " +
       "SET r.type = rel.dependencyType, " +

void saveDependencyRelations(List<Map<String, Object>> relations);

### 2.4 检索时的依存关系处理

在记忆检索时，需要考虑依存关系：

```java
private List<FactNode> retrieveContextWithDependencies(String userId, String message) {
    // 1. 基础检索
    List<FactNode> baseFacts = retrieveContext(userId, message);
    
    // 2. 检查是否有事实被标记为 conflicted/superseded
    boolean hasInvalidated = baseFacts.stream()
        .anyMatch(f -> "conflicted".equals(f.getStatus()) || "superseded".equals(f.getStatus()));
    
    if (hasInvalidated) {
        // 3. 过滤掉失效事实及其依赖链
        baseFacts = baseFacts.stream()
            .filter(f -> !"conflicted".equals(f.getStatus()) && !"superseded".equals(f.getStatus()))
            .collect(Collectors.toList());
        
        // 4. 记录过滤日志
        systemLogService.info("检索过滤: 已排除失效事实及其依赖链", "MEMORY");
    }
    
    return baseFacts;
}
```

## 三、其他优化建议

### 3.1 事实提取的并发安全

**问题**: 高频对话场景下，同一用户的多个 `extractFacts` 线程可能导致 Neo4j 节点的版本锁竞争。

**建议方案**:
- 引入基于 `userId` 的分布式锁
- 对同一用户的事实提取操作串行化
- 增加重试和降级策略

### 3.2 向量库与图谱的一致性保障

**问题**: 当前删除事实时，Neo4j 和 pgvector 的删除是分开执行的，存在不一致风险。

**建议方案**:
- 引入事务性保障（如 Saga 模式）
- 增加定期一致性校验任务（每日执行）
- 记录失败的重试队列

### 3.3 记忆生命周期维护的性能优化

**问题**: 当前 `runMemoryMaintenance()` 使用 `factRepository.findAll()` 加载所有事实到内存。

**建议方案**:
- 改为分批处理（Batch Processing，每批100条）
- 增加增量更新机制（仅处理 lastActivatedAt 在维护周期内的事实）
- 记录维护任务的执行时间趋势
