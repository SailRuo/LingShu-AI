# 后端实施细节 (Backend Implementation Details)

## 一、核心流程实现

### 1.1 事实提取链路 (Cognitive Fact Extraction)
实现类: `MemoryServiceImpl.extractFacts`

- **异步触发**: 使用 Spring `@Async` 在对话回复后立即触发，不阻塞用户感知。
- **情感门控**: 调用 `EmotionAwareFactExtractor` (LLM)，根据当前情绪强度调整提取策略。
- **时间归一化**: LLM 提示词内置时间上下文，自动将“昨天”、“上周”转换为 ISO-8601 绝对时间。
- **去重与冲突检测**: 
  - **精确匹配**: 归一化文本字符串比较。
  - **语义替代 (SUPERSEDES)**: 使用向量检索 + LLM 关系评估，识别并覆盖旧版本事实。
  - **冲突处理 (CONTRADICTS)**: 识别语义矛盾的事实，将旧事实标记为 `conflicted`。

### 1.2 情感片段存储 (Episodic Memory)
实现类: `EmotionalEpisodeService`

- **触发条件**: 情绪强度 > 0.5 且存在明确触发事件。
- **存储载体**: 存储在 Neo4j `EmotionalEpisode` 节点中，属性包含：`triggerEvent`, `copingMechanism`, `outcomeEmotion` 等。
- **关联用户**: 通过 `EXPERIENCED_BY` 关系链接至 `UserNode`。

### 1.3 记忆维护任务 (Maintenance Scheduler)
实现类: `MemoryMaintenanceScheduler` & `MemoryServiceImpl.runMemoryMaintenance`

- **执行周期**: 每小时执行一次 (`0 0 * * * *`)。
- **状态流转算法**:
  - `active` (>0.75): 6小时内有访问。
  - `stable` (0.3 - 0.75): 24小时至7天内访问。
  - `cool` (<0.3): 长期未访问。
  - `archived`: 虽然未被物理删除，但在 RAG 检索中权积极低，重要性系数 × 0.85。
- **孤立节点清理**: 自动清理无内容的冗余节点。

## 二、存储架构细节

### 2.1 Neo4j 图谱 (Knowledge Graph)
- **事实节点 (Fact)**: 存储核心内容及其元数据。
- **关系类型**:
  - `HAS_FACT`: 用户 -> 事实。
  - `SUPERSEDES`: 事实(新) -> 事实(旧)。
  - `CONTRADICTS`: 事实(新) -> 事实(旧)。
  - `RELATED_TO`: 事实 -> 事实。
- **索引优化**: 建立了 `originalMessage` 和 `emotionalTone` 的属性索引。

### 2.2 Pgvector 向量库 (Vector Store)
- **存储内容**: 格式化为 `用户原始消息 ||| 提取事实陈述`。
- **元数据保留**: 在向量 Metadata 中保留 `fact_id`, `category`, `confidence`, `volatile` 等标签，便于检索后的过滤。

## 三、待改进与技术债务

1. **逻辑依存关系 (Logical Dependency)**:
   - 目前尚未实现代码层面的 `DEPENDS_ON` 自动识别。
   - 计划在 `persistFactCandidateWithMetadata` 中引入新的关系决策分支。
2. **向量同步一致性**:
   - 当 Neo4j 节点被手动删除时，PGVector 的异步删除可能因网络波动失败，需增加每日定时的一致性检查任务。
3. **并发冲突**:
   - 高频对话场景下，同一用户的多个 `extractFacts` 线程可能导致 Neo4j 节点的版本锁竞争。
4. **批量操作 API (Batch Ops)**:
   - 增加 `DELETE /api/memory/fact/bulk` 支持前端多选删除。
   - 优化批量更新逻辑，确保 Neo4j 事务的一致性。
