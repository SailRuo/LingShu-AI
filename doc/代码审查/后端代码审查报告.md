## 一、安全问题

### 1. 数据库凭据硬编码
- **位置**：`lingshu-web/src/main/resources/application.yml:13-21`
- **描述**：PostgreSQL 和 Neo4j 的用户名、密码直接明文写在配置文件中
- **代码片段**：
  ```yaml
  datasource:
    url: jdbc:postgresql://localhost:5432/lingshu
    username: postgres
    password: lingshu123
  neo4j:
    uri: bolt://localhost:7687
    authentication:
      username: neo4j
      password: lingshu123
  ```

### 2. Redis 连接无认证配置
- **位置**：`lingshu-web/src/main/resources/application.yml:25-27`
- **描述**：Redis 配置仅指定了 host 和 port，未配置密码，若 Redis 启用了认证则会连接失败；若未启用认证则存在安全风险

### 3. 缺少用户权限校验
- **位置**：`lingshu-web/src/main/java/com/lingshu/ai/web/controller/ChatController.java:108-118`
- **描述**：`clearHistory` 接口仅通过 userId 参数操作，未校验请求者是否有权操作该用户的会话数据，存在越权风险
- **代码片段**：
  ```java
  public void clearHistory(@RequestParam(name = "userId", defaultValue = "User") String userId,
                           @RequestParam(name = "sessionId", required = false) Long sessionId) {
      Long effectiveSessionId = resolveSessionId(userId, sessionId);
      chatService.clearHistory(effectiveSessionId);
  }
  ```

### 4. API Key 可能通过日志泄露
- **位置**：`lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/ChatServiceImpl.java:257`
- **描述**：日志中打印了 apiKey 的值（即使做了 `***` 处理，但仍存在逻辑上的敏感信息日志输出）
- **代码片段**：
  ```java
  log.info("使用 OpenAI 兼容模式, effectiveUrl={}, model={}, apiKey={}, enableThinking={}", effectiveUrl, model, apiKey != null ? "***" : "null", enableThinking);
  ```

## 二、代码质量问题

### 5. 重复代码 - 图谱检索方法
- **位置**：`lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:649-681` 与 `734-779`
- **描述**：`performGraphRetrieval` 和 `performGraphRetrievalV2` 两个方法逻辑高度相似，存在大量重复代码
- **影响**：维护困难，修改一处需要同步修改另一处
- **优化方案**：
  1. 抽取统一模板方法 `performGraphRetrievalCore(GraphRetrievalContext ctx)`，将“候选收集、二跳扩展、去重、打分”沉淀为公共流程。
  2. 将 V1/V2 差异（阈值、排序权重、是否启用二跳）下沉到 `GraphRetrievalStrategy` 配置对象，避免复制整段逻辑。
  3. 增加参数化单元测试覆盖两套策略，确保重构后行为不回归。

### 6. 重复代码 - 向量检索方法
- **位置**：`lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:683-732` 与 `781-824`
- **描述**：`performVectorRetrieval` 和 `performVectorRetrievalV2` 两个方法逻辑高度相似，存在大量重复代码
- **优化方案**：
  1. 提炼统一入口 `performVectorRetrieval(VectorRetrievalRequest request)`，通过 `request.mode` 控制 V1/V2 行为。
  2. 将“嵌入召回 -> 过滤 -> 重排 -> 截断”拆成私有小函数，保证单函数单职责。
  3. 把阈值和 topK 提取到配置（如 `application.yml`）并加默认值，消除魔法数字分散。

### 7. 方法参数过多
- **位置**：`lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/ChatServiceImpl.java:190-191`
- **描述**：`streamChat` 方法签名包含 9 个参数，难以维护和使用
- **方法签名**：
  ```java
  public Flux<String> streamChat(String message, List<String> images, Long sessionId, Long agentId, String userId, String model, String apiKey, String baseUrl, Boolean enableThinking, ToolEventListener toolEventListener)
  ```
- **优化方案**：
  1. 新建不可变请求对象 `ChatStreamRequest`（builder 模式），统一承载 message/images/session/model 等参数。
  2. 将模型配置相关字段封装为子对象 `ModelOptions`，减少顶层参数数量。
  3. 保留旧签名一段时间作为兼容层，内部立即转为新对象，逐步迁移调用方。

### 8. 构造方法参数过多
- **位置**：`lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/ChatServiceImpl.java:72-106`
- **描述**：`ChatServiceImpl` 构造方法有 16 个参数，违反单一职责原则，难以维护
- **影响**：依赖注入复杂，单元测试困难
- **优化方案**：
  1. 按职责拆分协作组件：`ModelClientFactory`、`SessionCoordinator`、`StreamingResponseAssembler`、`ToolInvocationService`。
  2. 使用聚合依赖对象（如 `ChatRuntimeDependencies`）收敛基础设施类注入，降低构造参数个数。
  3. 为核心逻辑补充面向接口的单测，避免重构期间行为漂移。

### 9. 未实现的接口
- **位置**：`lingshu-web/src/main/java/com/lingshu/ai/web/controller/ChatController.java:34-37`
- **描述**：`/api/chat/welcome` 接口返回 `Flux.empty()`，未实现实际功能
- **代码片段**：
  ```java
  @GetMapping(value = "/welcome", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
  public Flux<String> getWelcome(@RequestParam(name = "userId", defaultValue = "User") String userId) {
      return Flux.empty();
  }
  ```
- **优化方案**：
  1. 若暂不提供功能：返回 `501 Not Implemented`（或统一业务错误码），并在接口文档标注未开放。
  2. 若计划保留：提供最小可用实现（欢迎文案 + 用户上下文摘要），避免前端空流等待。
  3. 增加契约测试，明确该接口的响应语义（空流/固定事件/错误码）并固化到 CI。
