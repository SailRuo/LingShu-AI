## 一、文档目的

本文档不再把“现状审计”“论文机制说明”“目标架构设计”“实施路线图”混写在一起，而是拆成三个问题分别回答：

1. **我们现在到底实现了什么**
2. **我们距离真正的 GAM-RAG 还差什么**
3. **应该按什么顺序演进，才能在当前工程里稳妥落地**

本文的重点不是复刻论文表述，而是为当前代码库提供一条**低风险、可验证、可回滚**的演进路径。

## 二、执行摘要

### 2.1 核心判断

当前系统已经具备 **“图谱检索 + 向量兜底 + 基于静态特征的检索路由”** 的雏形，但**尚未实现真正意义上的增益自适应记忆更新**。

更准确地说，当前实现属于：

- 已实现：**静态增益驱动的检索路由**
- 已实现：**基于共现次数的关系增强**
- 已实现：**被命中事实的标量活跃度刷新**
- 未实现：**答案后反馈闭环**
- 未实现：**基于反馈的自适应记忆更新**
- 未实现：**基于不确定性的增益调度**
- 未实现：**记忆向量沿查询方向的在线调整**

### 2.2 对论文落地的建议

不建议直接一步到位引入完整的 `MemoryState + Kalman + LLM Feedback + 向量持久化` 方案。  
更稳妥的路径是分三阶段推进：

1. **反馈闭环最小版**  
   先判断“哪些检索事实对回答真正有帮助”，但暂时只更新标量分数与关系权重。
2. **自适应排序版**  
   把反馈结果用于图谱排序、兜底决策和关系强化，验证检索收益。
3. **向量状态版**  
   在前两阶段收益稳定后，再引入记忆向量、不确定性与卡尔曼式更新。

这样做的原因很简单：当前代码中并不存在文档假设的“每条事实已具备可持久化的记忆状态向量”这一基础设施，直接跳到完整论文方案，工程风险过高。

### 2.3 对当前方案的定位

当前方案并不是“错误方向”，而是更适合被定义为：

> **GAM-RAG 的前置路由版（routing-oriented baseline）**

它解决的是：

- 图谱和向量检索如何协同
- 图谱结果何时值得优先使用
- 在召回率与噪声之间如何做静态平衡

但它还没有解决 GAM-RAG 最关键的问题：

> **回答结束后，系统如何判断哪些记忆真的帮到了回答，并据此持续修正后续检索。**

因此，当前实现可以作为后续增益自适应能力的 baseline，但不应被误认为已经完成了 GAM-RAG 的核心闭环。

## 三、当前实现审计

本节只描述**当前代码中已经存在的能力**，不混入未来设计。

### 3.1 已实现能力总览

| 能力 | 当前状态 | 说明 |
|------|----------|------|
| 图谱检索 | 已实现 | 基于实体命中激活事实，并支持二跳扩展 |
| 向量检索 | 已实现 | 图谱不足时执行语义召回 |
| 检索路由 | 已实现 | 依据静态 gain 决定图谱优先、图谱直出或向量兜底 |
| 检索事件记录 | 已实现 | 记录实体、命中事实、最终内容、路由结果等 |
| 关系增强 | 已实现 | 基于共现计数写入 `RELATED_TO` 权重 |
| 事实刷新 | 已实现 | 被命中事实提升 `importance/confidence/activityScore` |
| 检索后反馈闭环 | 未实现 | 没有判断“事实是否真正帮助回答” |
| 自适应状态更新 | 未实现 | 没有 uncertainty、gain scheduling、directional update |

### 3.2 检索主流程现状

当前主流程位于：

- [MemoryServiceImpl.java](/E:/Project/LingShu-AI/backend/lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:583)

实际执行逻辑可概括为：

```text
用户消息
  -> 实体提取
  -> 图谱检索
  -> 静态 gain 计算
  -> 路由判断
     - gain 高：图谱优先 + 向量补充
     - gain 低且图谱空：向量兜底
     - gain 低且图谱非空：图谱直出
  -> 合并去重
  -> 记录检索事件
  -> 异步更新关系共现
```

这里的关键点是：**gain 只参与“检索策略决策”，不参与“检索后记忆学习”**。

### 3.3 当前 gain 的本质

静态增益计算位于：

- [MemoryServiceImpl.java](/E:/Project/LingShu-AI/backend/lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:822)

现有公式本质上是：

```text
gain = 实体匹配率 × 0.6 + 平均重要性 × 0.4
```

并对以下情况做惩罚：

- `superseded` 事实
- 90 天前的陈旧事实
- 二跳事实

因此，当前 gain 更准确的定义应是：

> **“图谱结果在检索前看起来有多可信”**

而不是：

> **“系统是否已经从回答效果中学会该如何检索”**

### 3.4 当前图谱检索方式

图谱检索位于：

- [MemoryServiceImpl.java](/E:/Project/LingShu-AI/backend/lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:675)

其核心特征：

- 一跳事实主要靠**实体字符串命中**
- 命中的事实按 `importance` 排序
- 结果不足时补充核心事实
- 一跳命中较少时继续做二跳扩展

这说明当前系统的图谱能力仍然是：

- **基于规则激活**
- **基于标量排序**
- **基于局部关系扩展**

还不是“基于可学习记忆状态的检索”。

### 3.5 当前关系增强方式

关系增强位于：

- [MemoryServiceImpl.java](/E:/Project/LingShu-AI/backend/lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:863)

当前逻辑是：

1. 读取一次检索事件中被纳入上下文的事实
2. 统计这些事实对在历史事件中的共现次数
3. 共现次数达到阈值后，建立或增强 `RELATED_TO`

这属于一种**轻量级共现强化机制**，而不是真正的增益自适应学习。它的问题主要有两个：

- 它把“被放进上下文”近似为“对回答有帮助”
- 它只会正向增强，不会因为噪声或无效命中而保守回退

### 3.6 当前事实刷新方式

事实刷新位于：

- [MemoryServiceImpl.java](/E:/Project/LingShu-AI/backend/lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java:1650)

当前被命中的事实会：

- 更新 `normalizedContent`
- 更新 `lastActivatedAt`
- 刷新 `activityScore`
- 将 `importance` 抬升到至少 `0.82`
- 将 `confidence` 抬升到至少 `0.88`
- 将 `cool` 状态恢复为 `active`

这套机制解决了“事实被反复命中时不应该持续冷却”的问题，但也带来一个明显副作用：

> **只要被检索到，就会被提升；至于它有没有真正帮助回答，系统并不关心。**

### 3.7 现有数据结构边界

检索事件结构位于：

- [MemoryRetrievalEvent.java](/E:/Project/LingShu-AI/backend/lingshu-core/src/main/java/com/lingshu/ai/core/dto/MemoryRetrievalEvent.java:1)

事实节点结构位于：

- [FactNode.java](/E:/Project/LingShu-AI/backend/lingshu-infrastructure/src/main/java/com/lingshu/ai/infrastructure/entity/FactNode.java:1)

当前 `FactNode` 已有：

- 内容
- 类别
- 观测时间 / 事件时间 / 最后激活时间
- `importance`
- `confidence`
- `activityScore`
- `status`
- `ttlDays`
- `version`
- 替代 / 冲突链路

但**没有**以下 GAM-RAG 所需状态：

- 任务记忆向量
- 时序记忆向量
- 不确定性估计
- 更新次数
- 记忆状态版本

因此，如果要实现真正的增益自适应，**必须先补状态模型与存储边界**。

## 四、与真正 GAM-RAG 的差距

本节只讨论“差距”，不直接展开实现细节。

### 4.1 缺失一：没有答案后反馈闭环

论文类 GAM-RAG 方案成立的前提是：

```text
检索 -> 生成回答 -> 评估哪些检索项真的有用 -> 更新记忆
```

而当前系统只有：

```text
检索 -> 生成回答 -> 结束
```

缺了这一环，就无法回答下面这些核心问题：

- 哪些事实只是“看起来相关”，但实际上没有被答案使用
- 哪些事实每次都能稳定支撑回答
- 哪些图谱激活是噪声，应该降权而不是增强

### 4.2 缺失二：没有“被采用事实”的真实定义

当前 `adoptedFactIds` 更接近：

> **被纳入上下文的事实**

而不等同于：

> **被回答真正采用的事实**

这是一个关键概念差异。如果后续把“进入上下文”直接拿来做学习信号，系统会放大伪相关噪声。

因此，真正的反馈闭环必须引入新的定义：

- `retrievedFacts`：被检索出来的事实
- `contextFacts`：被放进 prompt 的事实
- `supportedFacts`：被答案真正使用的事实

只有第三类才能作为高质量正反馈。

### 4.2.1 `supportedFacts` 的推荐定义

建议将 `supportedFacts` 定义为：

> **在最近一轮问答中，被模型回答实际依赖、引用或用于推理支撑的事实集合。**

它和下面两类概念必须严格区分：

- `retrievedFacts`：检索阶段曾命中的事实
- `contextFacts`：被放入 prompt 的事实

判断某条事实是否属于 `supportedFacts`，不应只看“它是否被放进上下文”，而应看它是否满足以下至少一类条件：

1. **直接引用**
   回答中明确使用了该事实中的核心信息。
2. **关键支撑**
   回答虽然没有原文复述，但结论显然依赖了该事实。
3. **背景约束**
   该事实限定了回答的语气、偏好、对象或时序条件，使回答更贴合用户上下文。

以下情况不应算作 `supportedFacts`：

- 事实被检索到了，但回答完全没用到
- 事实与 query 表面相关，但只是噪声陪跑
- 事实出现在上下文中，但没有对答案内容产生可辨识影响

### 4.3 缺失三：没有不确定性建模

论文中最有价值的部分，不是“用了向量”，而是：

> **让系统知道自己对某条记忆有多确定，因此决定应该学得快还是学得慢。**

当前系统没有这个能力，所以它无法区分：

- 新出现、尚不稳定的记忆
- 多次验证、非常稳定的记忆
- 可能错误但暂时看起来相关的记忆

### 4.4 缺失四：没有负向反馈的保守更新机制

当前系统只有“命中即增强”的正向偏置，没有处理：

- 检索到了但没用
- 多次检索都无用
- 与正确答案相冲突
- 被更新后的新事实取代

如果未来引入在线学习，却没有负向反馈的保守机制，系统很容易被偶然共现或 LLM 误判污染。

### 4.5 缺失五：没有分阶段验证策略

文档原方案默认直接引入：

- 反馈分析器
- 记忆状态向量
- Neo4j 持久化
- 卡尔曼增益更新
- 检索排序联动

这个跨度过大，问题在于当效果不好时，很难判断究竟是：

- 反馈信号不准
- 更新公式有问题
- 状态存储不稳定
- 排序逻辑放大了噪声

所以工程上必须拆阶段推进。

## 五、目标设计原则

在当前 LingShu-AI 代码库中落地增益自适应时，建议遵循以下原则。

### 5.1 先做闭环，再做复杂状态

先证明系统能稳定判断“哪些检索结果对回答有效”，再讨论复杂状态更新。  
没有可靠反馈信号时，任何在线学习都只是扩大噪声。

### 5.2 先改排序与决策，再改底层数据模型

如果只是想先验证收益，可以先把反馈信号用于：

- 图谱结果排序
- 向量兜底触发阈值
- `RELATED_TO` 权重更新
- `importance/confidence` 的精细化更新

这样能更快拿到收益，也能避免过早大规模改库表和持久化结构。

### 5.3 强化“可观测性优先”

任何自适应机制上线前，都必须先能观察：

- 哪些事实被检索
- 哪些事实进入上下文
- 哪些事实被答案采用
- 哪些事实在更新后提升了后续召回质量

没有观测指标，后续调参会非常盲。

### 5.4 默认异步，默认可回滚

反馈分析和状态更新都不应阻塞主回答链路。  
同时每个阶段都要允许通过开关关闭，退回当前静态路由策略。

### 5.5 优先兼容现有 `FactNode`

除非进入第三阶段，否则不建议一开始就把 `FactNode` 升级成重型向量状态实体。  
前两阶段尽量在现有字段和附属状态表之上完成。

## 六、建议的三阶段演进路线

### 6.1 阶段一：反馈闭环最小版

#### 目标

建立“检索结果是否真正帮助回答”的判断闭环，但暂时不引入记忆向量和卡尔曼更新。

#### 这一阶段新增的能力

- 记录一次回答对应的检索上下文快照
- 在回答完成后评估哪些事实被真正使用
- 将“被采用”与“仅被检索到”区分开
- 为后续优化提供日志和指标

#### 建议实现方式

新增一个轻量反馈分析组件，输出如下结构：

```json
{
  "query": "...",
  "factFeedback": [
    {
      "factId": 123,
      "usedInAnswer": true,
      "confidence": 0.82,
      "reason": "答案直接引用了该事实"
    }
  ]
}
```

这一阶段可以接受两种实现路线：

- **规则优先**：通过答案文本和事实内容做引用匹配或语义对齐
- **LLM 辅助**：让模型判断事实是否真正支撑回答

建议从“规则优先 + LLM 兜底”开始，原因是更便于控制成本与误判。

#### `supportedFacts` 的判定建议

推荐按“最近一轮问答”做反馈判断，也就是：

- 输入：`当前 query + 当前轮检索上下文 + 当前 assistant response`
- 输出：当前轮哪些事实属于 `supportedFacts`

其中，LLM 适合承担的是：

> **对最近一轮对话中的候选事实逐条判断：该事实是否为本轮回答的有效事实。**

这是目前最接近 GAM-RAG 精神、也最容易在现有工程里落地的做法。

建议判定过程分两层：

##### 第一层：规则粗筛

先用廉价规则过滤明显无效或明显有效的情况，例如：

- 答案中是否直接出现事实关键词
- 答案是否复述了事实中的关键信息
- 事实与 query/response 的语义相似度是否过低

规则层的作用不是最终定案，而是：

- 过滤明显噪声
- 减少 LLM 判定成本
- 为 LLM 提供更干净的候选集合

##### 第二层：LLM 逐条判定

对规则层保留下来的候选事实，由 LLM 针对最近一轮问答做逐条判断：

- `valid=true`：该事实对本轮回答构成有效支撑
- `valid=false`：该事实虽被检索或进入上下文，但未对回答形成有效帮助

建议 LLM 输出至少包含：

```json
{
  "factId": 123,
  "valid": true,
  "confidence": 0.84,
  "reason": "答案直接使用了该事实中的用户偏好信息"
}
```

其中 `confidence` 很重要，因为后续不应该把所有 `valid=false` 都视为强负反馈。  
很多情况下，LLM 只能做到“低置信无效”，这时更适合：

- 冻结分数
- 轻微降权
- 标记待观察

而不是立刻做强烈负向更新。

#### 为什么只看最近一轮

建议第一版只判断最近一轮，而不是跨多轮回看，原因有三点：

1. **时序最清晰**  
   当前 query、当前上下文、当前回答三者因果关系最明确。
2. **工程实现最稳**  
   可以用 `turnId/requestId` 直接绑定一次检索与一次回答。
3. **反馈噪声更低**  
   跨轮回看容易把历史上下文、续写内容和当前回答混在一起。

后续如果确实需要，再考虑加入多轮累积反馈，但第一版不建议一开始就做复杂。

#### 本阶段暂不做的事

- 不更新向量状态
- 不引入 uncertainty
- 不修改图谱主排序公式
- 不改 Neo4j 数据模型

#### 验收标准

- 能稳定记录“检索事实 / 上下文事实 / 支撑事实”三层数据
- 能抽样复核反馈结果，误判率可控
- 不影响主回答链路延迟

### 6.2 阶段二：自适应排序版

#### 目标

在不引入复杂向量状态的前提下，让系统根据真实反馈优化下一轮检索质量。

#### 这一阶段新增的能力

- 反馈结果驱动 `importance/confidence` 精细更新
- 反馈结果驱动 `RELATED_TO` 权重增减
- 反馈结果参与图谱排序和向量兜底决策
- 引入“无效命中”的保守负反馈

#### 建议更新策略

可先使用简化版标量更新，而不是直接上卡尔曼：

```text
若事实被答案采用：
  importance 小幅上调
  confidence 小幅上调
  relation weight 适度增强

若事实进入上下文但未被答案采用：
  importance 轻微下调或冻结
  confidence 保守下降
  relation weight 不增强
```

这一步的价值在于先验证：

- 反馈是否真的能提升排序质量
- 负反馈是否会造成错误抑制
- 哪类问题更适合图谱优先，哪类问题仍应依赖向量兜底

#### 验收标准

- 离线回放集上的上下文命中质量优于当前基线
- 常见问题的无效图谱命中率下降
- 关系增强不再只会单向累加

### 6.3 阶段三：向量状态版

#### 目标

在前两阶段反馈链路和收益已被验证的前提下，引入真正的增益自适应状态模型。

#### 这一阶段才建议新增的核心状态

```text
MemoryState
- factId
- taskVector
- timeVector
- taskUncertainty
- timeUncertainty
- updateCount
- lastUpdate
- stateVersion
```

#### 为什么要放到第三阶段

因为这一阶段会同时引入多个高风险变量：

- 新状态模型
- 新持久化结构
- 查询向量方向更新
- 不确定性衰减
- 超参数调优

如果前两阶段都还没证明“反馈信号可用”，那这一步只会把复杂度快速拉高。

#### 进入该阶段的前提

- 已有可复用的高质量反馈数据
- 已有明确的离线评估集
- 已经知道哪些问题是当前排序逻辑的瓶颈
- 已有回滚开关，可退回阶段二逻辑

## 七、目标架构建议

本节描述建议中的目标形态，但默认按阶段渐进落地。

### 7.1 关键概念重定义

后续实现中，建议明确区分四类对象：

| 概念 | 含义 |
|------|------|
| `retrievedFacts` | 检索阶段命中的所有事实 |
| `contextFacts` | 实际组装进 prompt 的事实 |
| `supportedFacts` | 被答案真正使用的事实 |
| `memoryState` | 该事实跨轮次累积的可学习状态 |

这四类概念混在一起，会直接导致反馈信号失真。

其中 `supportedFacts` 在第一版中的推荐判定口径为：

- 观察窗口：仅最近一轮问答
- 判定主体：规则粗筛 + LLM 判定
- 判定结果：`valid / confidence / reason`
- 用途：作为高质量正反馈和弱负反馈的主要来源

### 7.2 建议的数据流

```text
用户问题
  -> 实体提取 / 图谱检索 / 向量检索
  -> 结果合并与路由决策
  -> 组装上下文
  -> 生成回答
  -> 反馈分析
  -> 记忆更新
  -> 指标记录
```

其中“反馈分析”和“记忆更新”建议异步执行，避免阻塞主链路。

### 7.3 阶段一到二的建议组件

建议先引入以下轻量组件：

#### `RetrievalContextSnapshot`

职责：

- 保存一次回答前的检索上下文
- 关联 query、retrieved facts、context facts、timestamp

#### `RetrievalFeedbackAnalyzer`

职责：

- 输入 query、context facts、assistant response
- 输出每条事实是否真正支撑回答

#### `AdaptiveMemoryScorer`

职责：

- 根据反馈结果更新标量评分
- 供后续图谱排序与关系更新使用

### 7.4 阶段三的建议组件

在阶段三再考虑新增：

- `MemoryState`
- `MemoryStateRepository`
- `KalmanGainUpdater`
- `MemoryStateProjector`

这几类组件应该建立在“阶段一、二已经证明反馈有效”的前提下。

## 八、阶段三的卡尔曼式更新建议

本节保留论文对应机制，但明确其位置是**第三阶段目标**，不是当前立刻实现项。

### 8.1 适合保留的论文思想

真正值得引入的，不是论文中的每个形式化定义，而是下面三条思想：

1. **正反馈学得快，负反馈学得慢**
2. **越不确定的记忆，更新幅度越大**
3. **更新沿查询方向进行，而不是全局乱漂移**

### 8.2 进入工程实现前必须先回答的问题

在真正实现卡尔曼式更新前，必须先明确：

1. `taskVector` 初值从哪里来  
   当前系统并没有可直接复用的事实级持久向量状态。
2. `timeVector` 是否第一版就要上  
   建议不要。先只做 `taskVector`。
3. 反馈标签是否足够稳定  
   如果 `usedInAnswer` 判定本身波动大，卡尔曼更新会被错误信号污染。
4. 状态存储放在哪里  
   是 Neo4j 节点属性、旁路存储，还是附属状态仓库。

### 8.3 第三阶段的建议最小实现

如果未来进入该阶段，建议最小实现只包含：

- `taskVector`
- `taskUncertainty`
- `updateCount`
- 正负反馈非对称噪声

先不要同时引入：

- `timeVector`
- PCA 可视化
- 复杂多头记忆
- 额外 GPU 优化

## 九、建议的里程碑与验收口径

相比“按组件拆工时”，更建议使用“按收益与可验证性拆里程碑”。

### M1：反馈数据可用

**目标**

系统能可靠地区分：

- 检索到了什么
- 放进上下文了什么
- 真正帮助回答了什么

**验收**

- 有完整日志链路
- 有抽样复核方法
- 有基础 dashboard 或统计指标

### M2：反馈可改善排序

**目标**

反馈结果能稳定改善图谱排序或兜底决策。

**验收**

- 离线集优于现网基线
- 无效事实进入上下文的比例下降
- 误伤核心事实的情况可控

### M3：状态模型可离线验证

**目标**

引入 `MemoryState` 后，先在离线或灰度环境验证其收益。

**验收**

- 状态写入稳定
- 参数变动对结果的影响可观察
- 可以一键回退到 M2 逻辑

### M4：在线灰度自适应更新

**目标**

小流量启用真正的增益自适应更新。

**验收**

- 主链路延迟无明显恶化
- 记忆污染率可控
- 检索质量指标持续优于基线

## 十、主要风险与应对

### 10.1 反馈信号失真

**风险**

如果“是否被答案采用”的判断本身不准，后续所有自适应学习都会被带偏。

**应对**

- 阶段一先做人工抽样复核
- 规则优先，LLM 辅助
- 为反馈结果增加置信度，而不是直接二值化硬写回

### 10.2 进入上下文不等于有效支撑

**风险**

如果直接用 `adoptedFactIds` 作为正反馈，会持续强化伪相关事实。

**应对**

- 明确定义 `supportedFacts`
- 关系增强与分数提升仅对高置信支撑事实生效

### 10.3 异步更新与会话时序错位

**风险**

回答完成后异步回写，可能拿错最近一次检索上下文。

**应对**

- 用 requestId / turnId 绑定检索与回答
- 不要只按“最近一次 userId 上下文”去关联

### 10.4 多实例缓存一致性

**风险**

若后续引入内存态 `MemoryState` 缓存，多实例部署下容易出现状态不一致。

**应对**

- 前两阶段避免依赖本地内存态作为唯一真相
- 第三阶段再评估是否引入缓存层

### 10.5 过早复杂化

**风险**

一开始就引入向量状态、卡尔曼增益、可视化和多种超参数，会让调试成本失控。

**应对**

- 严格按三阶段推进
- 每阶段先证明收益，再继续扩展

## 十二、结论

### 12.1 对当前系统的定性

当前 LingShu-AI 已经实现了 **“带静态增益判断的混合检索”**，但还没有实现 **“检索后基于回答效果持续学习的记忆系统”**。

### 12.2 最重要的演进判断

真正的瓶颈不是“还没有卡尔曼公式”，而是：

> **系统还不知道哪些检索结果真的帮到了回答。**

在这个问题解决前，直接引入复杂在线更新机制，收益不确定，风险却很高。

### 12.3 推荐的下一步

推荐按以下顺序推进：

1. 先补“检索事实是否真正支撑回答”的反馈闭环
2. 再用反馈优化现有标量排序与关系增强
3. 最后在收益明确后引入向量状态与增益调度

这条路线更符合当前代码基线，也更适合逐步验证 GAM-RAG 在本项目中的真实价值。
