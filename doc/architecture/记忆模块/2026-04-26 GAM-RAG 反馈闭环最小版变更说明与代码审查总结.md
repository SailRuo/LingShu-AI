## 二、执行摘要

本次改动已经将当前记忆系统从“**仅有静态路由判断**”推进到“**具备最近一轮反馈闭环，并且能把反馈回写到排序相关标量与关系权重**”。

更具体地说，本次已经完成：

- 按 `turnId` 保存一次检索的上下文快照
- 在回答结束后，针对最近一轮 `contextFacts` 判断哪些事实真正支持了回答
- 采用“规则优先 + LLM 兜底”的方式输出 `valid / confidence / reason`
- 将事实级反馈持久化为 `RetrievalFeedbackRecord`
- 用 `supportedFacts` 更新 `FactNode.importance/confidence`
- 用高置信 `supportedFacts` 更新 `RELATED_TO.weight`
- 将更新后的 `importance/confidence/relationWeight` 接入图谱排序与 `gain`
- 在回合后处理中记录反馈汇总日志，并清理快照，避免残留

本次**仍然没有**实现：

- `MemoryState / uncertainty / Kalman` 风格的向量状态更新
- 基于不确定性的增益调度

因此，本次实现的准确定位应为：

> **GAM-RAG 的阶段一反馈闭环 + 阶段二标量自适应排序版**

而不是完整的阶段三向量状态学习系统。

## 三、本次改动说明

### 3.1 新增回合级检索快照

本次新增了与检索反馈相关的领域对象与接口：

- `RetrievalContextSnapshot`
- `RetrievalFactCandidate`
- `RetrievalFeedbackResult`
- `RetrievalContextSnapshotStore`
- `RetrievalFeedbackAnalyzer`

这些对象的作用是把“一次检索发生时的上下文”稳定保存下来，供回答完成后进行反馈判断。

快照中包含的关键信息包括：

- `userId`
- `sessionId`
- `turnId`
- `query`
- `routingDecision`
- `gain`
- `retrievedFacts`
- `contextFacts`

其中最关键的是 `contextFacts`，因为阶段一反馈闭环只需要判断：

> **进入本轮上下文的事实里，哪些真的支撑了回答。**

### 3.2 新增快照捕获与会话清理

本次将检索快照接入了记忆检索链路：

- 在检索阶段，如果存在 `turnId`，则保存本轮检索快照
- `contextFacts` 不再通过渲染后的字符串反推，而是直接来自实际被选入上下文的事实集合
- 在清空会话历史时，会同步删除该会话关联的检索快照

这一步保证了反馈分析所依赖的数据来源是“**真实参与了上下文构造的事实**”，而不是后置猜测或字符串还原。

### 3.3 新增 `supportedFacts` 判定器

本次实现了 `RetrievalFeedbackAnalyzerImpl`，用于对最近一轮上下文事实进行逐条判断。

当前策略为：

- 第一层：规则直判  
  如果事实内容与助手回复存在明显直接匹配，则直接判为有效支持
- 第二层：LLM 兜底  
  对于规则无法稳定判断的事实，交由 LLM 输出：
  - `valid`
  - `confidence`
  - `reason`

当前判定对象只限定在：

- **最近一轮**
- **本轮 `contextFacts`**

这样做的目的是降低时序混淆，避免把“历史上提过但本轮没实际使用”的事实误判为有效支持。

### 3.4 新增回合后反馈服务

本次新增 `RetrievalFeedbackService`，负责在回答完成后执行以下动作：

1. 按 `turnId` 读取检索快照
2. 调用 `RetrievalFeedbackAnalyzer`
3. 汇总 `supported / unsupported / uncertain` 统计
4. 写入系统日志
5. 删除本轮快照

这样做有两个好处：

- 把“分析逻辑”和“回合后处理主流程”解耦
- 即使后续情感分析、事实提取等流程提前返回或异常，检索快照也能被及时消费和清理

### 3.5 将 `turnId` 接入回合后处理链路

本次把 `turnId` 从聊天主链路一路传递到回合后处理入口，使反馈分析与具体回答回合绑定。

接线后，回合后处理顺序变为：

1. 先执行检索反馈分析
2. 再继续既有的情感分析、事实提取、互动记录等后处理逻辑

这个顺序是刻意设计的，因为检索反馈只依赖：

- 当前轮 query
- 当前轮 contextFacts
- 当前轮 assistant response

它不应该被后续其他后处理分支是否成功所阻塞。

## 四、与完整 GAM-RAG 的关系

### 4.1 本次已经具备的部分

从 GAM-RAG 视角看，本次已经补上了一个关键前提：

> **回答后，系统可以开始判断哪些事实真的对本轮回答有帮助。**

这意味着系统第一次具备了“检索结果质量的回合后观测能力”。

### 4.2 本次尚未具备的部分

不过，本次实现仍停留在“标量反馈学习层”，尚未进入“向量状态学习层”。

当前还缺少：

- 更系统的多轮负反馈抑制策略
- `uncertainty`、状态向量与增益调度
- Kalman 风格的更新幅度控制

因此，本次结果不能定义为“已经完整实现 GAM-RAG”，更合适的定义是：

> **完成了 GAM-RAG 阶段一最小闭环，并落地了阶段二的标量自适应排序路径。**

## 五、代码审查结论

### 5.1 结论

本次实现未发现新的阻断性问题，已经满足当前 plan 的目标边界。

已确认的正向结论包括：

- 回合级快照的数据结构已建立
- `turnId` 贯穿了检索与后处理链路
- `RetrievalFeedbackAnalyzer` 具备规则优先与 LLM 兜底能力
- 分析完成后快照会被删除，避免无界残留
- 会话清空时会同步清理快照
- 聚焦测试已通过

### 5.2 当前剩余风险

虽然没有阻断问题，但当前仍有几个明确风险点：

1. **LLM 判定结果仍可能抖动**  
   规则直判降低了一部分噪声，但兜底 LLM 仍可能在边界样例上不稳定。

2. **仍是最近一轮闭环，不是多轮归因系统**  
   这对当前阶段是合理的，但更复杂的长期学习仍需要更稳定的跨轮归因模型。

3. **关系负反馈目前是保守降权，不是结构化关系重写**  
   当前 `unsupportedFacts` 只对已有 `RELATED_TO` 做小步降权，不会创建负向关系边，也不会跨主题大幅重排图结构。

## 八、后续建议

结合当前已落地状态，建议顺序更新为：

1. 先观察数据库唯一约束与并发补偿在真实流量下的冲突率，必要时补重试退避与告警
2. 持续观察关系负反馈降权是否存在误伤，按离线回放结果校准阈值与步长
3. 持续观察阶段三指标面板（`stateUpdates / stateBonus / uncertainty`）并做阈值校准
4. 在指标稳定后，再评估是否进入 `Kalman / 增益调度 / timeVector` 扩展

这样可以把当前已经建立的“反馈驱动自适应闭环”，进一步收敛成“可长期稳定运行”的版本。
