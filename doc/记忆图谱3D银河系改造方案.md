# 灵枢记忆图谱 3D 银河系改造方案

## 1. 背景

当前灵枢的“记忆图谱”已经具备基础可视化能力，能够展示用户节点与事实节点之间的关系，但整体表现仍然偏向传统的 2D 力导向图，视觉冲击力、空间层次感、信息表达力都还有较大提升空间。

从产品气质上看，灵枢不是一个普通的后台管理系统，而是一个强调“生命感”“陪伴感”“白盒感知”的本地 AI 生命体。记忆图谱理应承担以下职责：

1. 让用户感受到“记忆在生长”，而不只是“数据被存进数据库”。
2. 让用户直观看懂哪些记忆是核心、哪些是边缘、哪些正在活跃。
3. 让系统的长期记忆机制可检查、可追踪、可纠偏。
4. 让图谱界面本身成为产品记忆系统的“主舞台”，而不是附属工具页。

因此，将现有 2D 图谱升级为 3D 银河系式记忆宇宙，是一个合理且值得推进的方向。

---

## 2. 结论

结论先行：

**可以做，而且适合做。**

但要明确三个前提：

1. 这不是简单“换个皮肤”，而是一次前端渲染层和图谱数据表达层的联合升级。
2. 如果继续基于当前 `Neovis.js + vis-network` 的 2D 思路强行堆 3D 视觉，会出现“看起来像银河，但交互和结构仍然很平”的问题。
3. 若想真正做出“银河系记忆宇宙”的质感，必须同步优化后端图谱数据结构，否则最终只会变成“一个中心点拖着一圈散点”的伪 3D 效果。

换句话说：

**3D 可以做，但必须和记忆结构重构一起做，效果才会成立。**

---

## 3. 当前实现现状

### 3.1 前端图谱现状

当前前端图谱页面位于：

- `frontend/src/views/InsightView.vue`

主要特征：

1. 通过 `Neovis.js` 直接连接 Neo4j。
2. 图形渲染本质仍是 2D 网络图。
3. 页面已经有“星云背景、光晕、网格、漂浮层”等视觉包装。
4. 工具栏、右侧详情面板、节点点击/右键删除等基础交互已具备。

目前已有的优点：

1. 氛围感已经有方向，不是纯后台图表风格。
2. 节点高亮、详情侧栏、删除事实这些交互都已经跑通。
3. 页面视觉语言和灵枢的整体品牌气质并不冲突。

目前明显不足：

1. 图谱本体仍然是平面网络，缺少真正的纵深和轨道结构。
2. 中文长文本直接作为节点标签，不适合大规模图谱浏览。
3. 节点布局逻辑仍然是通用力导向，不具备“星系秩序感”。
4. 统计面板中的 `density / nodes / edges / latency` 目前是静态展示，不是真实指标。
5. 图谱页直接携带 Neo4j 连接信息，存在明显安全问题。

### 3.2 后端图数据现状

当前后端图谱接口位于：

- `backend/lingshu-web/src/main/java/com/lingshu/ai/web/controller/MemoryController.java`

对应服务实现位于：

- `backend/lingshu-core/src/main/java/com/lingshu/ai/core/service/impl/MemoryServiceImpl.java`

当前图数据结构本质是：

1. `User`
2. `Fact`
3. `User -[HAS_FACT]-> Fact`

也就是说，当前图谱基本是一个中心用户节点，外加一圈事实节点。

这对“功能演示”是足够的，但对“银河系式记忆宇宙”来说还远远不够，因为银河结构需要：

1. 核心层
2. 轨道层
3. 主题簇
4. 事件带
5. 活跃区与冷区
6. 不同类型节点之间的层级关系

而当前图谱还没有这些层次。

## 4. 当前系统存在的问题

这一部分分为两类：

1. 图谱可视化层问题
2. 记忆机制本身的问题

### 4.1 图谱可视化层问题

#### 问题 1：前端直接连接 Neo4j

当前 `InsightView.vue` 中存在 Neo4j 连接配置，包含：

1. `serverUrl`
2. `serverUser`
3. `serverPassword`

这是一个必须优先修复的问题。

风险包括：

1. 数据库连接信息暴露到前端。
2. 线上部署时无法可靠控制数据库访问策略。
3. 无法在后端做鉴权、限流、过滤、裁剪、脱敏。
4. 图谱展现逻辑与数据库结构强耦合，后续很难自由演化。

结论：

**无论是否做 3D，这一项都应尽快改为“前端只请求后端图谱 API”。**

#### 问题 2：视觉是星云包装，结构仍是普通图

当前页面已经有较强的星云背景和氛围层，但节点结构仍是平面力导向图，所以用户看到的是：

1. 背景像宇宙
2. 节点却还是普通关系图

这会产生一种“外壳很美，但核心没跟上”的割裂感。

#### 问题 3：标签密度不可扩展

当前事实节点直接显示全文，后期记忆一旦增长，会迅速出现：

1. 标签重叠
2. 画面拥挤
3. 节点过密导致可读性下降
4. 缩放后要么全看不清，要么全糊成一团

银河系式 3D 可视化通常要采用分层显示策略：

1. 远景只显示星点和轨道
2. 中景显示简短标签
3. 近景才展示完整文本或详情面板

#### 问题 4：缺少“记忆活动”表达

现在的图谱更像静态地图，缺少动态感知：

1. 哪些记忆刚刚被写入
2. 哪些记忆刚被检索命中
3. 哪些记忆最近活跃
4. 哪些记忆即将衰减

如果做银河系风格，这些都可以直接映射成：

1. 星点亮度
2. 轨道流光
3. 脉冲波纹
4. 尘埃拖尾
5. 温度色彩变化

### 4.2 记忆机制本身的问题

#### 问题 1：当前图谱结构过于扁平

目前主要是 `User -> Fact` 单层关系。

这样会导致：

1. 图谱没有多跳语义结构。
2. 无法表达“事实之间的关系”。
3. 无法形成“主题星团”。
4. 无法实现真正的图谱导航，只能看一圈事实点。

建议未来扩展为：

1. `User`
2. `Fact`
3. `Topic`
4. `Event`
5. `EmotionState`
6. `Preference`
7. `Project`
8. `Person`
9. `Place`
10. `TimeAnchor`

以及关系：

1. `HAS_FACT`
2. `BELONGS_TO_TOPIC`
3. `RELATED_TO`
4. `CAUSED_BY`
5. `AFFECTS_EMOTION`
6. `MENTIONED_WITH`
7. `HAPPENED_AT`
8. `SUPERSEDES`
9. `CONTRADICTS`

#### 问题 2：检索上下文拼接方式容易膨胀

当前 `retrieveContext` 的逻辑，是先把用户全部事实拼进上下文，再决定要不要做语义检索。

这会带来几个问题：

1. 用户事实多了以后，prompt 会越来越大。
2. 不相关事实也会进入上下文，增加模型干扰。
3. 真正有用的记忆和无关的旧事实没有优先级差异。

更合理的做法应该是：

1. 先根据 query 做意图和主题识别。
2. 再从结构化图谱中召回候选集合。
3. 再做向量检索和重排序。
4. 最后只把最相关的一小组记忆注入 prompt。

#### 问题 3：事实去重过于粗糙

当前去重主要依赖“事实文本完全相等”。

这会导致：

1. 同义表达无法合并。
2. 细微措辞变化会产生重复记忆。
3. 图谱会慢慢出现大量语义重复节点。

例如：

1. “我喜欢咖啡”
2. “我平时很爱喝咖啡”
3. “咖啡是我的最爱”

这三句本质很接近，但目前可能会被存成多条事实。

建议改为混合去重：

1. 文本标准化去重
2. 向量相似度去重
3. 事实类型去重
4. 冲突合并与版本更新

#### 问题 4：实体提取较粗糙

当前 `extractEntities` 逻辑更接近“分词 + 停用词过滤”，对中文复杂表达支持有限。

容易出现的问题：

1. 对口语句式不稳定。
2. 对多义词处理能力弱。
3. 对跨句指代能力弱。
4. 对时间语义不敏感。
5. 对关系语义不敏感。

例如：

1. “上次那个项目”
2. “她之前说的那件事”
3. “最近又开始焦虑了”

这些表达都不是简单关键词匹配就能处理好的。

#### 问题 5：记忆缺少生命周期管理

当前事实写入后，主要依靠 `importance` 字段，但并没有一套完整的生命周期机制。

缺少的能力包括：

1. 记忆衰减
2. 激活加权
3. 冷记忆冻结
4. 冲突替换
5. 阶段性记忆归档
6. 过期事件记忆压缩

换句话说，现在更像“把事实存进去”，还不是“让记忆活起来”。

#### 问题 6：图谱接口没有真正成为前端唯一数据源

虽然已有 `/api/memory/graph`，但前端并未围绕该接口构建。

这会导致：

1. 前后端职责不清晰。
2. 图谱结构升级时，需要直接改前端 Cypher。
3. 前端无法做版本兼容。
4. 后端无法对图谱节点做业务级裁剪和聚合。

## 5. 3D 银河系方案目标

如果要做，我建议这个页面的目标不是“把图做成 3D”，而是做成：

**可交互的记忆宇宙。**

它至少应具备以下特征：

### 5.1 视觉目标

1. 用户节点是银河核心或恒星核心。
2. 主题类记忆围绕核心形成轨道带。
3. 事实节点是星点、星团或小型卫星体。
4. 活跃记忆具有脉冲和流光效果。
5. 刚写入的记忆以“生成尘埃”或“新星闪现”方式出现。
6. 被检索命中的记忆显示高亮引力线。
7. 长期沉寂的记忆进入暗区或冷区。

### 5.2 交互目标

1. 支持旋转、缩放、平移。
2. 支持聚焦某一主题簇。
3. 支持按时间、重要性、类型筛选。
4. 支持点击节点进入记忆详情。
5. 支持查看某条记忆的来源、首次出现时间、最近激活时间、置信度。
6. 支持删除、归档、合并、修正记忆。

### 5.3 信息表达目标

3D 不只是炫技，而是用空间表达结构：

1. 距离表达语义亲近度。
2. 大小表达重要性。
3. 颜色表达记忆类型。
4. 亮度表达近期活跃度。
5. 轨道层级表达主题归属。
6. 动态流动表达检索与写入过程。

## 6. 推荐技术方案

### 6.1 不推荐继续沿用的方案

不建议继续把 `Neovis.js` 当成主渲染方案来实现 3D 银河系。

原因：

1. 它适合图数据库快速展示，但不适合做高定制宇宙场景。
2. 真正的 3D 空间感、粒子效果、轨道动画、景深控制都不擅长。
3. 中文文本、节点层次、性能控制会越来越难做。

### 6.2 推荐方案 A：Three.js 定制 3D 场景

推荐程度：最高

适用场景：

1. 想做强视觉风格
2. 想把“银河系”做成产品级核心页面
3. 后续还想加粒子、流光、路径回放、时序动画

优点：

1. 可控性最高
2. 视觉表现上限最高
3. 可做真正的宇宙场景和深度交互
4. 适合构建“灵枢独有”的审美语言

缺点：

1. 开发成本较高
2. 需要手写更多节点布局与交互逻辑
3. 需要更严格的性能治理

### 6.3 推荐方案 B：3d-force-graph / force-graph-3d

推荐程度：中高

适用场景：

1. 希望较快做出 3D 图谱
2. 想保留图结构布局能力
3. 希望在较短周期内验证交互价值

优点：

1. 开发速度快
2. 比纯 Three.js 更省布局开发成本
3. 能较快从现有图结构迁移过来

缺点：

1. 视觉上限不如纯 Three.js
2. 深度定制能力有限
3. 做“银河系秩序感”时容易仍然像“3D 力导向图”

### 6.4 建议选择

建议采用：

**第一阶段用 `3d-force-graph` 快速验证交互和结构表达，第二阶段逐步过渡到 `Three.js` 定制宇宙场景。**

这样风险最小。

## 7. 推荐的数据模型升级

如果要让 3D 银河系效果成立，后端图数据不能只输出节点和边的基本列表，而应该输出更适合可视化语义分层的数据结构。

建议输出模型至少包含以下字段：

### 7.1 节点字段建议

```json
{
  "id": "fact_123",
  "label": "用户最近在准备考研",
  "type": "Fact",
  "subType": "Goal",
  "importance": 0.86,
  "confidence": 0.91,
  "activityScore": 0.72,
  "cluster": "career_growth",
  "orbitLevel": 2,
  "createdAt": "2026-03-25T18:30:00",
  "lastActivatedAt": "2026-03-25T19:10:00",
  "sourceTurnId": "chat_889",
  "status": "active"
}
```

### 7.2 边字段建议

```json
{
  "source": "topic_career_growth",
  "target": "fact_123",
  "type": "BELONGS_TO_TOPIC",
  "weight": 0.78,
  "lastActivatedAt": "2026-03-25T19:10:00"
}
```

### 7.3 节点分层建议

建议分成四层：

1. 核心层
   - `User`
2. 主题轨道层
   - `Topic`
   - `Project`
   - `EmotionDomain`
3. 记忆实体层
   - `Fact`
   - `Preference`
   - `Person`
   - `Event`
4. 时间与状态层
   - `TimeAnchor`
   - `State`

### 7.4 可视化映射建议

1. `importance` 映射节点大小
2. `activityScore` 映射亮度和呼吸频率
3. `cluster` 决定星团归属
4. `orbitLevel` 决定离核心的层级距离
5. `confidence` 决定边缘发光强度
6. `status` 决定颜色风格

## 8. 3D 银河系视觉设计建议

### 8.1 总体视觉概念

建议将整个图谱视为一个“记忆星系”：

1. 核心恒星：用户自我画像
2. 第一轨道：稳定身份与核心偏好
3. 第二轨道：长期项目、关系、重要人物
4. 第三轨道：近期事件、情绪变化、阶段目标
5. 外围尘埃：低权重、低活跃记忆

### 8.2 节点视觉规则

#### User 节点

1. 作为中央高亮恒星
2. 持续缓慢脉冲
3. 带核心光晕和引力场

#### Topic 节点

1. 作为星团中心或轨道锚点
2. 颜色稳定且识别度高
3. 可作为聚焦入口

#### Fact 节点

1. 作为普通星点或碎片卫星
2. 重要事实更大更亮
3. 冷记忆更暗、透明度更低

#### Event 节点

1. 可采用拖尾粒子
2. 支持按时间展开回放

### 8.3 动画建议

1. 页面进入时先显示星尘聚合，随后形成星系结构。
2. 新记忆写入时从外围尘埃聚拢成星点。
3. 记忆被检索命中时，沿边缘显示一次流光。
4. 聚焦某个主题簇时，相机缓慢推进并压暗非目标区域。
5. 删除记忆时节点不直接消失，而是碎裂并淡出。

### 8.4 文本与标签策略

1. 默认仅显示核心节点和高重要性节点标签。
2. 远景显示简短标题，不显示全文。
3. 近景或 hover 时再显示完整信息。
4. 详情永远在侧栏展开，不在 3D 空间里堆长文本。

## 9. 记忆机制优化建议

这一部分是重点，因为视觉升级如果没有记忆质量提升，会很快失去意义。

### 9.1 记忆写入优化

建议将当前“事实提取”升级为“记忆写入决策 + 去重合并 + 生命周期赋值”三段式流程。

#### 第一段：写入决策

判断这条信息是否值得写入长期记忆。

考虑因素：

1. 是否稳定
2. 是否可复用
3. 是否与身份、偏好、项目、关系、情绪状态有关
4. 是否只是短期噪声

#### 第二段：语义去重与合并

将新事实与已有记忆对比：

1. 完全重复则跳过
2. 高相似度则合并
3. 冲突事实则更新旧节点并记录版本关系
4. 阶段性变化则建立时间版本链

#### 第三段：生命周期赋值

写入时就赋予：

1. `importance`
2. `confidence`
3. `activityScore`
4. `memoryType`
5. `ttl`
6. `decayRate`

### 9.2 记忆检索优化

建议把当前检索改为四步：

1. Query 意图分类
2. 图谱候选召回
3. 语义向量召回
4. 融合重排序

可增加：

1. 时间权重
2. 活跃度权重
3. 用户显式提及权重
4. 当前情绪上下文权重

### 9.3 图谱结构优化

建议逐步加入“主题簇”概念。

即：

1. 事实不直接全部挂在用户身上
2. 事实先归入主题
3. 主题再围绕用户构成轨道

这样可以天然支撑银河系视觉。

例如：

1. `用户 -> 职业发展 -> 考研 -> 目标院校`
2. `用户 -> 情绪健康 -> 焦虑 -> 最近睡眠差`
3. `用户 -> 兴趣偏好 -> 咖啡 -> 手冲 -> 某品牌豆子`

### 9.4 时间机制优化

当前系统里时间信息还没有成为图谱核心能力。

建议加入：

1. 首次出现时间
2. 最近激活时间
3. 最近确认时间
4. 衰减时间
5. 阶段有效期

这样可以支持：

1. 时间回溯
2. 记忆升温/降温
3. 某一阶段的人生状态回放

### 9.5 记忆质量治理

建议建立后台治理能力：

1. 重复记忆检测
2. 孤立节点检测
3. 冲突事实检测
4. 长期未激活记忆归档
5. 低置信度记忆审查

## 12. 风险与注意事项

### 12.1 性能风险

3D 图谱页面比当前 2D 页面更吃性能，尤其在以下场景：

1. 节点过多
2. 文本标签过密
3. 粒子效果太多
4. 动画过重

建议：

1. 节点数量做分层加载
2. 标签按距离显示
3. 远景使用 instancing 或简化点渲染
4. 大图谱按簇懒加载

### 12.2 信息过载风险

银河系效果容易好看，但如果没有清晰的信息分层，用户会只觉得“酷”，却看不懂。

因此必须坚持：

1. 场景为信息服务
2. 动效为状态表达服务
3. 标签和详情分离

### 12.3 数据质量风险

如果后端记忆质量不稳定，3D 会把问题放大：

1. 重复事实会变成重复星点
2. 噪声事实会污染轨道结构
3. 冲突事实会让画面和认知都变乱

所以 3D 之前，至少要先解决基础去重和主题归类问题。

---

## 13. 最终建议

综合判断，建议如下：

### 可以做

3D 银河系风格非常适合灵枢，方向正确，而且会显著提升产品辨识度。

### 但不要只改前端皮肤

如果只在现有 2D 图上套 3D 外观，最终效果会比较虚。

### 应该按“数据先行、渲染升级、交互增强”推进

最佳路径是：

1. 先清理当前前端直连 Neo4j 的问题
2. 再升级后端图谱数据模型
3. 再引入 3D 渲染层
4. 最后做高级动效和记忆宇宙叙事
