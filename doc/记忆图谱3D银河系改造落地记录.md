# 记忆图谱 3D 银河系改造落地记录

本文档用于记录截至当前版本，`记忆图谱 3D 银河系改造方案` 已经落地的内容、具体实现位置、验证情况，以及尚未彻底完成的事项。

3. 图谱结构已经从扁平的 `User -> Fact`，升级为带主题层的 `User -> Topic -> Fact`。
4. 事实节点已经不只是纯文本，而是开始具备生命周期、活跃度、版本、冲突、替代等真实记忆属性。
5. 前端页面已经具备“银河系式记忆宇宙”的核心交互能力，包括：
   - 主题轨道
   - 活跃度筛选
   - 时间筛选
   - 时间回放
   - 检索命中流光
   - 新记忆高亮
   - 新写入闪现
   - 版本替代/冲突链路显示

换句话说：

**这套改造已经从“视觉概念验证”进入“可运行的机制化实现”。**

### 这些字段当前承担的职责

1. `normalizedContent`
   - 用于文本规范化去重。
2. `subType`
   - 表达更细粒度的事实类型，如 `Preference / Goal / Event / EmotionState`。
3. `clusterKey`
   - 用于主题归类与前端星域映射。
4. `lastActivatedAt`
   - 支撑活跃度计算和时间回放。
5. `activityScore`
   - 支撑前端高活跃/稳定/冷区筛选。
6. `status`
   - 区分 `active / stable / cool / superseded / conflicted`。
7. `version`
   - 用于表达替代链中的版本演进。
8. `supersedesFactId / contradictsFactId`
   - 记录替代和冲突关系。

### 落地价值

这一步非常关键，因为它意味着：

**图谱不再只是“展示已有文本”，而是在数据层面开始具备“记忆”应有的演化属性。**

## 3.4 事实写入链路已具备去重、替代、冲突处理

### 已完成内容

当前事实写入逻辑已经不再是“只要是新字符串就保存”。

现在的写入策略大致为：

1. 先做文本规范化。
2. 检查是否命中精确去重。
3. 如果命中精确去重，则不新建事实，而是刷新旧事实的：
   - `lastActivatedAt`
   - `activityScore`
   - `importance`
   - `confidence`
   - `status`
4. 如果与已有事实相似度高，则创建新事实，并将旧事实标为 `superseded`。
5. 如果与已有事实存在冲突，则创建冲突版本，并将旧事实标为 `conflicted`。
6. 如果既不重复也不冲突，则创建新的活跃事实。

### 当前后端已经支持的真实机制

1. 精确去重
2. 相似替代
3. 冲突版本创建
4. 活跃度刷新
5. 版本号记录
6. 新增事实向量化写入 embedding store

### 现阶段局限

当前“相似替代”和“冲突检测”仍然属于启发式推断，不是最优方案，但已经是真实的后端行为，而不是前端展示时临时猜出来的。

## 3.5 图谱关系已支持关联链、替代链、冲突链

### 已完成内容

后端图谱接口现在会额外推断事实之间的关系：

1. `RELATED_TO`
2. `SUPERSEDES`
3. `CONTRADICTS`

### 当前推断来源

后端会按：

1. 同一用户
2. 同一主题簇

对事实进行分组，然后依据：

1. 词法相似度
2. 否定/肯定的冲突特征
3. 观察时间先后

生成关系。

### 当前语义

1. `RELATED_TO`
   - 语义相近，但不足以判定为版本替代。
2. `SUPERSEDES`
   - 新事实可视为旧事实的后续版本。
3. `CONTRADICTS`
   - 新旧事实在语义上出现冲突。

### 落地价值

这一步让图谱开始具备“记忆变化轨迹”的表达能力，而不是只会展示静态节点。

## 5. 当前仍未彻底完成的事项

这里的“未完成”不是指完全没做，而是指还没有达到方案里的“彻底落地”标准。

## 5.1 Neo4j 中的真实关系模型已完成 P0 落地

### 当前状态

以下关系已经真正作为 Neo4j 边写入图数据库，并在写入链路和图谱输出链路中维护：

1. `(:Fact)-[:RELATED_TO]->(:Fact)`
2. `(:Fact)-[:SUPERSEDES]->(:Fact)`
3. `(:Fact)-[:CONTRADICTS]->(:Fact)`

当前做法是：

1. 新事实写入或旧事实刷新后
2. 后端按同用户、同主题簇的事实集合重新同步关系
3. 删除旧的事实关系边
4. 重建最新的 `RELATED_TO / SUPERSEDES / CONTRADICTS`
5. 图谱 API 读取阶段只加载 Neo4j 中已持久化的真实关系，不再在读接口里推断并写回关系
6. 删除事实节点、记忆生命周期维护完成后，也会重新同步剩余事实关系，保证关系图不会因为删除或归档而失真

### 已落地价值

1. 图谱读取开始依赖真实关系，而不是只依赖临时推断
2. 版本链导航有了持久化基础
3. 冲突关系成为真实图数据
4. 为后续做图数据库级查询和记忆治理打下基础
5. 图谱读取链路已经做到“读真实关系、写走维护链路”，避免读取接口带副作用

## 5.2 主题归类仍然是启发式规则，不够稳定

### 当前状态

Topic 和 subType 目前主要依赖：

1. `category`
2. 文本关键词启发式规则
3. 新写入事实已经开始接入“历史相似事实复用 + 规则预分类 + LLM 语义兜底”的混合分类链路
4. 记忆维护任务在发现 `clusterKey / subType` 缺失或过于泛化时，也会补齐主题轨道与子类型
5. `category` 与 `subType` 已从混用状态调整为优先存储主题键和事实子类型分离

### 当前问题

1. 混合策略虽然已经落地，但历史主题复用目前仍然基于轻量词法相似度，不是向量级召回。
2. LLM 兜底已经接入，但还没有做系统性的分类评估与回放验证。
3. 主题归类和子类型分类仍然缺少人工纠偏闭环。
4. 复杂口语化、多义句、跨句依赖场景下，分类稳定性还有继续优化空间。

### 彻底落地建议

后续应升级为混合策略：

1. 规则预分类
2. LLM 语义分类
3. 历史主题簇匹配
4. 再做回写和纠偏

当前已完成前 3 步的基础闭环，下一阶段更应该补的是：

1. 向量召回参与历史主题簇匹配
2. 分类结果来源与置信度可观测
3. 人工纠偏/反馈回写链路

## 5.4 记忆生命周期调度已完成第一阶段落地

### 当前状态

现在后端已经具备真实的记忆维护调度能力：

1. 每小时自动执行一次记忆维护任务
2. 可通过 API 手动触发维护
3. 可通过 API 查看最近一次维护摘要

当前维护任务已经会做：

1. 刷新 `activityScore`
2. 补全缺失的 `normalizedContent / subType / clusterKey`
3. 修正 `confidence / version / decayRate / ttlDays`
4. 将低活跃记忆降为 `cool`
5. 将重新活跃的记忆恢复为 `active`
6. 将过期且低活跃或已替代的记忆归档为 `archived`

### 当前图谱行为

已归档记忆默认不会进入图谱可视化结果。

### 当前接口

1. `POST /api/memory/maintenance/run`
2. `GET /api/memory/maintenance/summary`

### 仍未完成的更深层能力

这一块还没有完全做到最终形态，仍缺：

1. 低置信度记忆审查流
2. 归档后独立冷库/历史区
3. 更细的分层回收策略
4. 前端维护结果可视化面板

## 5.6 真正的 Three.js 宇宙场景仍未开始

### 当前状态

当前前端已经做出“银河视觉语言”和“轨道布局表达”，但本质上仍然是 DOM/SVG 驱动的 2.5D 方案，不是完整 Three.js 场景。

### 这不代表当前方案无效

**先验证结构表达和交互价值。**

### 彻底落地建议

如果后续继续推进：

1. 可以保留当前 API、语义字段、筛选逻辑不变
2. 将渲染层逐步替换为 `Three.js`
3. 复用现在的：
   - Topic 轨道概念
   - activityScore
   - orbitLevel
   - clusterKey
   - relation type

也就是说，当前阶段做的是“数据与交互地基”，未来可以自然迁移到更强的 3D 渲染层。
