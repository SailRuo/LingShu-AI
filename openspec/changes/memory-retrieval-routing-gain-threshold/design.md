## Context

当前记忆检索采用“先图谱、再向量”的组合策略，但路由条件不够清晰，导致两类体验问题并存：图谱命中很好时，向量补召回不稳定；图谱命中较弱时，又容易因为过早跳过向量而丢失可用上下文。与此同时，向量检索目前缺少足够明确的用户范围约束，容易把不相关记忆带入当前上下文。

这次设计的目标不是重做检索系统，而是把已有图谱和向量两条链路收敛成一套可解释、可观测、可回归的路由策略。

## Goals / Non-Goals

**Goals:**
- 建立基于图谱召回质量的检索路由规则。
- 在图谱质量较高时保留向量补召回能力。
- 在图谱质量较低或命中不足时提供可控的向量兜底。
- 让图谱结果与向量结果能稳定合并、去重，并保持图谱优先。
- 缩小向量检索的噪声范围，避免跨用户结果污染当前上下文。

**Non-Goals:**
- 不重构事实抽取、分类或写入链路。
- 不重新设计记忆图谱的数据模型。
- 不在本次变更中调优大模型提示词或语义分类模型。

## Decisions

1. **使用 gain 作为主路由信号**
   - 选择已有的 `gain` 计算结果作为图谱召回质量判断依据，因为它已经同时考虑了实体覆盖和事实重要度，能够直接复用现有逻辑。
   - 备选方案是引入全新的质量评分模型，但这会增加额外训练/调参成本，也不利于快速回归。

2. **图谱高质量时执行“图谱优先 + 向量补召回”**
   - 当 `gain` 达到阈值时，图谱结果先进入上下文，随后补充少量向量结果用于覆盖长尾信息。
   - 这样可以保住图谱的高精度，同时利用向量补足图谱漏召回。
   - 备选方案是始终并行执行两路检索，但这会增加成本，并且在图谱很差时更容易把噪声引入上下文。

3. **图谱低质量时采用“有结果则直接返回，无结果再向量兜底”**
   - 当图谱结果已经足够提供上下文时，不再为了“试一下”而额外发起向量查询，避免无效开销。
   - 当图谱结果为空或不可用时，允许向量检索作为兜底，避免召回完全失效。
   - 备选方案是低质量时一律不查向量，但这会明显恶化召回覆盖率，不符合记忆检索场景的目标。

4. **向量检索必须带用户范围约束**
   - 向量检索结果必须限定在当前用户或当前会话可见范围内，避免跨用户内容混入上下文。
   - 现有 embedding 元数据需要补齐可过滤字段，并在查询时加上过滤条件。
   - 备选方案是继续全局检索再事后过滤，但这会浪费检索预算，并且会放大噪声。

5. **合并策略采用图谱优先、按事实去重**
   - 图谱结果保留在前，向量结果作为补充。
   - 如果两路命中的是同一事实，则只保留一条，避免重复上下文浪费 token。
   - 备选方案是按分数统一排序，但这会削弱图谱优先的业务意图，也更难解释。

## Risks / Trade-offs

- [阈值固定] → `0.3` 可能需要后续根据真实召回数据调整，当前先以默认阈值落地并通过日志观察。
- [用户过滤依赖元数据] → 旧 embedding 记录如果没有补齐过滤字段，需要在本地项目阶段直接重建或批量回填，避免检索混入无关事实。
- [补召回引入额外成本] → 高质量图谱结果会触发额外向量查询，需通过 `maxResults`、超时和过滤条件控制开销。
- [去重策略影响排序] → 统一去重后，少量边缘结果的顺序可能变化，需要通过回归测试稳定输出。

## Migration Plan

1. 先为 embedding 写入补齐用户范围元数据，保证新写入数据可过滤。
2. 在检索侧增加基于 `gain` 的路由逻辑，并保留清晰日志，便于对比新旧行为。
3. 为旧数据准备一次性回填或重建方案，确保所有可检索数据都带有用户范围元数据。
4. 上线后通过检索事件与命中率日志观察阈值效果，再决定是否把阈值外置为可配置项。
5. 如出现明显回退，可先关闭向量补召回路径，仅保留图谱结果和兜底路径。

## Open Questions

- `0.3` 是否应继续作为默认阈值，还是后续改成可配置项？
- 低增益且图谱有少量结果时，向量兜底的最大结果数应该是多少？
- 旧 embedding 的用户过滤字段已决定不做灰度兼容，是否需要一次性重建全部本地数据？
