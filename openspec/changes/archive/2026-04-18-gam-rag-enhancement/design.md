## Context

当前灵枢的记忆检索系统已经具备了基于 Gain 的路由能力（`GAIN_THRESHOLD = 0.3`），但图谱中的事实关系（`RELATED_TO`）仅基于词汇相似度（`lexicalSimilarity >= 0.55`）建立。这导致语义相关但字面不相似的事实之间无法建立关联（如"用户喜欢喝茶"与"用户喜欢读书"），图谱检索仅停留在单跳实体匹配阶段。

现有代码基础：
- `MemoryServiceImpl` 中的 `performGraphRetrieval` 负责图谱检索。
- `calculateGain` 计算图谱召回质量。
- `synchronizeFactRelations` 和 `guessFactRelationship` 负责建立事实关系，但仅使用词汇相似度。
- `FactRepository` 已提供 `saveRelatedRelations` 方法用于保存 `RELATED_TO` 关系。
- `MemoryRetrievalEvent` DTO 已定义检索事件结构，但缺少"采纳事实"字段。

## Goals / Non-Goals

**Goals:**
- 实现基于检索反馈的图谱关系动态建立机制，让共同被采纳的事实之间建立 `RELATED_TO` 关系。
- 在图谱检索阶段引入多跳激活逻辑，利用已有关系进行扩散检索。
- 优化 Gain 计算，引入时序惩罚和状态过滤。
- 补充检索反馈记录能力，为关系更新提供信号。

**Non-Goals:**
- 不修改底层 Embedding 向量（不实现论文中的卡尔曼滤波向量更新）。
- 不重写实体抽取实现。
- 不重构 Neo4j 数据模型或事实写入链路。
- 不在本次变更中引入 LLM 驱动的关系判断（仍使用词汇相似度 + 共现反馈）。

## Decisions

1. **使用共现反馈建立关系，而非 LLM 实时判断**
   - 选择基于检索事件中的"共同采纳"信号来建立关系，因为这种方式成本低、可异步执行。
   - 备选方案是每次新事实写入时调用 `FactRelationshipEvaluator`（LLM）判断与所有旧事实的关系，但这会导致写入延迟显著增加。
   - 决策：在 `MemoryRetrievalEvent` 中增加 `adoptedFactIds` 字段，检索完成后异步分析共现对并更新关系。

2. **多跳检索限制为最多 2 跳**
   - 选择 2 跳作为上限，因为 3 跳及以上容易引入过多噪声，且收益递减。
   - 第一跳：实体匹配直接命中的事实。
   - 第二跳：沿 `RELATED_TO` 关系扩散的相关事实，按关系权重排序，最多取 5 个。
   - 备选方案是动态决定跳数，但这会增加复杂度，先以固定 2 跳落地。

3. **Gain 计算引入时序和状态惩罚**
   - 对于 `status = "superseded"` 的事实，重要度贡献乘以 0.3。
   - 对于 `observedAt` 超过 90 天的事实，重要度贡献乘以 0.7。
   - 这些惩罚系数先以硬编码常量落地，后续可通过日志观察调整。

4. **关系更新异步执行**
   - 关系更新操作通过 `CompletableFuture.runAsync` 在独立线程池中执行，不阻塞主检索流程。
   - 使用现有的 `taskExecutor` 线程池，避免创建新线程池。

## Risks / Trade-offs

- [共现信号噪声] → 两个事实可能偶然被同时采纳，但不一定真正相关。通过设置最低共现次数（如 2 次）才建立关系来缓解。
- [多跳检索延迟] → 二跳检索需要额外查询 Neo4j，可能增加检索延迟。通过限制二跳最大结果数（5 个）和关系权重阈值（>= 0.5）控制。
- [关系爆炸] → 随着事实增多，`RELATED_TO` 关系数量可能快速增长。通过关系权重上限和定期清理低权重关系（< 0.3）来管理。
- [Gain 惩罚系数固定] → 0.3 和 0.7 的惩罚系数可能需要根据真实数据调整，当前先以默认值落地并通过日志观察。

## Migration Plan

1. 先在 `MemoryRetrievalEvent` 中增加 `adoptedFactIds` 字段，开始记录采纳信号。
2. 实现异步关系更新逻辑，利用已有共现信号建立关系。
3. 在 `performGraphRetrieval` 中增加二跳扩散逻辑。
4. 修改 `calculateGain` 增加时序和状态惩罚。
5. 上线后通过系统日志观测关系建立情况和多跳检索效果。

## Open Questions

- 共现多少次才建立关系？当前建议 2 次，但是否需要根据事实总数动态调整？
- 二跳检索的最大结果数 5 是否合适？是否需要根据一跳结果数量动态调整？
- 是否需要在前端增加"记忆关系可视化"功能，让用户看到事实之间的关联？
