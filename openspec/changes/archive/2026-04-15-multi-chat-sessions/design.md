## Context

当前灵墟之境的聊天数据已经按 `chat_sessions -> chat_turns -> chat_turn_events -> chat_turn_artifacts` 组织，但业务入口仍然把“用户”当成唯一对话边界：后端会优先取同一用户的第一条会话，前端也只维护一份消息流，没有会话列表和切换状态。

这意味着底层已经具备多会话的数据基础，缺口主要在会话管理层、接口层和对话页状态层。此次变更的目标不是重做聊天引擎，而是把“会话”正式提升为一级业务对象，让创建、切换、历史加载和清空都显式作用于当前会话。

## Goals / Non-Goals

**Goals:**
- 支持创建多个独立聊天会话。
- 支持查看会话列表并切换当前会话。
- 让聊天发送、历史加载、清空历史严格作用于当前会话。
- 保留旧数据和旧入口的兼容性，避免既有历史不可访问。
- 尽量复用现有 `chat_sessions` 及 turn/event 存储结构，降低迁移成本。

**Non-Goals:**
- 不做账号体系重构。
- 不做会话共享、协作编辑或跨用户转发。
- 不在本次变更中引入复杂的标签、搜索、归档或文件夹管理。
- 不重构 LLM 对话核心，只调整会话边界和路由方式。

## Decisions

1. **将 `sessionId` 设为聊天上下文的唯一主键。**
   - 选择原因：现有数据库和 turn timeline 已经按 `sessionId` 组织，沿用这一层最自然，风险也最低。
   - 备选方案：继续只用 `userId` 作为会话键。
   - 不选原因：`userId` 只能表达“谁在用”，不能表达“同时存在多少条对话”。

2. **保留现有表结构，优先通过接口和服务层补齐会话管理。**
   - 选择原因：`chat_sessions`、`chat_turns`、`chat_turn_events`、`chat_turn_artifacts` 已经足以支撑多会话；先补能力比做迁移更稳。
   - 备选方案：重新设计一套会话/消息模型。
   - 不选原因：会放大迁移和回归成本，也会让现有历史重新映射。

3. **前端显式维护当前会话状态，并将其持久化到本地。**
   - 选择原因：用户刷新页面后应回到上次使用的会话，避免每次都重新定位。
   - 备选方案：完全依赖后端按“最近一条会话”回退。
   - 不选原因：这种方式对多设备和并发会话不稳定，也会让切换行为不可预测。

4. **聊天请求、历史请求和清空请求都必须携带显式会话标识。**
   - 选择原因：这是避免“新建了会话，但消息又被写回旧会话”的关键手段。
   - 备选方案：仅在后端用 `userId` 自动推断当前会话。
   - 不选原因：推断逻辑无法稳定表达“当前会话”，尤其在多标签页或多窗口场景里。

5. **保留兼容回退逻辑，但只作为兜底，不作为主路径。**
   - 选择原因：旧客户端或旧历史入口不能瞬间失效。
   - 备选方案：直接移除所有无 `sessionId` 的调用。
   - 不选原因：会造成升级期兼容风险，尤其是已经存在的历史数据和自动化入口。

## Risks / Trade-offs

- [风险] 会话列表和当前会话状态需要前后端同步，容易出现“显示的是 A，发送落到 B”的错位。
  - [缓解] 所有聊天请求都显式带 `sessionId`，并在切换会话时同步刷新历史与消息流。
- [风险] 旧用户数据里可能只有单一默认会话，切换到多会话后会出现空白列表。
  - [缓解] 首次进入时自动创建默认会话，或把最早/最近会话作为回退项。
- [风险] 本地持久化的当前会话 ID 可能在会话被删除后失效。
  - [缓解] 启动时校验本地 `sessionId` 是否仍存在，不存在则降级为最近会话或新建会话。
- [风险] 同一用户在多个窗口同时操作时，当前会话状态可能互相覆盖。
  - [缓解] 把“当前会话”限定为每个浏览器实例的本地状态，而不是全局用户状态。

## Migration Plan

1. 先上线会话列表与创建接口，但保持旧的 `userId` 兜底逻辑不变。
2. 前端引入当前会话状态后，先默认选中最近一次会话，再逐步开放新建和切换入口。
3. 所有历史查询、发送消息和清空历史切到 `sessionId` 后，再把 `userId` 回退路径降级为兼容逻辑。
4. 观察一段时间后，如果旧兜底路径不再需要，再考虑收紧无 `sessionId` 请求的行为。

## Open Questions

- 会话是否需要支持重命名和删除？当前建议先不进入本次范围，等基础多会话跑稳后再评估。
- 新会话默认标题应该是什么？是固定“新对话”，还是根据首条消息自动摘要生成？
- 历史页面是否需要展示会话列表的时间顺序、标题和最后更新时间，还是只保留当前会话切换入口？
